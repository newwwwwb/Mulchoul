## 0. 가장 먼저 할 일
1. `_docs/물놀이시설_작업분배_웹앱_명세서.md` 를 처음부터 끝까지 정독하라. 이것이 요구사항의 절대 기준이다.
2. `_docs/물놀이근무보드_프로토타입.jsx` 를 정독하라. 이것은 **검증된 동작하는 프로토타입**이다. UI 구조, 역할 계산 로직(`rolesForTime`, `weekendRolesForTime`, `buildTimeSlots`, `classifyDate`, 공휴일 목록, 시간표 상수)은 이미 정확하므로 **그대로 이식**하라. 새로 발명하지 마라.
3. 두 파일 사이에 모순이 있으면 **명세서를 우선**하되, 모순 내용을 먼저 나에게 보고하고 진행하라.

## 1. 기술 스택 (명세서 11장 — 변경 금지)
- 프론트엔드: **React + Vite** (TypeScript 권장)
- DB/백엔드: **Supabase (PostgreSQL)**. 별도 API 서버 없이 프론트에서 supabase-js로 직접 접근
- 실시간 동기화: **Supabase Realtime** (출석·배정·바쁜 타임)
- 일일 배정 스케줄러: **Supabase pg_cron** (매일 자정 KST 자동 생성)
- 배포 대상: **Vercel**
- 상태 관리: React 기본(useState/Context). 외부 상태 라이브러리 금지.

## 2. 절대 지켜야 할 핵심 규칙 (명세서에서 확정된 사항 — 임의 변경 금지)

### 인원·식별
- 직원 7명 고정. 권한 구분 없이 **전원 동등**.
- 로그인/비밀번호 없음. **이름 = ID**. 출석 화면에서 이름 클릭이 본인 식별.

### 역할 계산 (프로토타입 로직 그대로)
- 평일(월~금): 3회차 = 6타임. 회차당 운영 110분 = 50분+휴식10분+50분.
  - A조(1·2·3)/B조(4·5·6)는 회차마다 문진↔현장 스위칭.
  - 매 타임 1→2→3→4→5→6 순서로 1명 휴식. **7번은 안 쉬고** 쉬는 사람의 조 역할을 대행.
  - 모든 타임 현장 3 + 문진 3 유지.
- 주말·공휴일(토·일·공휴일): 4회차 = 8타임.
  - 하루 **정확히 1개**의 "바쁜 타임"(기본 T3, 변경 가능): 전원 근무, 7번이 **문진 보강** → 현장 3 + 문진 4, 휴식 없음.
  - 나머지 7타임: 1→7번 순서로 1명씩 휴식(주말엔 7번도 1번 쉼). 현장 3 + 문진 3.
  - 바쁜 타임 위치에 따라 휴식 순서가 자동으로 밀린다.

### 날짜·시간대
- **월~금 = 평일, 토·일·공휴일 = 주말 운영.** 날짜로 자동 판별(토글 없음).
- 공휴일: 프로토타입의 2026년 한국 공휴일 목록을 `holidays` 테이블로 시드.
- **모든 시각·날짜·스케줄러는 Asia/Seoul(KST) 기준.** Postgres/cron 기본 UTC와 9시간 어긋나지 않게 명시적으로 변환하라.

### 출석·결근
- 이름 클릭 = 출석 + 시각 기록. **다시 누르면 출석만 취소(토글).** 대타 예약은 유지.
- 출석 안 한 번호는 "미출석" 표시. 운영 시작 후 눌러도 출석으로 전환(지각 허용).
- 결근 시 자리 메우는 규칙 없음. 역할 계산은 번호 기준 그대로, 화면엔 "미출석"만.

### 일일 배정
- 매일 자정(KST) pg_cron으로 자동 생성. 그날 행이 없으면 번호 배정·firstRoundARole **완전 랜덤** 생성, 주말이면 busy_time 기본 3 세팅.
- 개장(09:00) 전에도 그날 배정을 미리 볼 수 있어야 한다.

### 대타
- 대타는 별도 계정 아님. 결근 직원의 번호를 그대로 이어받음.
- 캘린더에서 "날짜 + 맡기는 직원 + 대타 이름" 사전 등록.
- 당일 대타가 맡긴 직원 이름을 눌러 출석 → 이름 뒤 `(대타)` 표시.
- 날짜/맡긴 직원/대타 이름 이력 조회 가능.

### 바쁜 타임 동기화
- 누구나 변경 가능. 변경 시 `daily_records.busy_time` 갱신 → Realtime으로 전원 즉시 반영.
- 대시보드에 현재 값 + **마지막 변경자·시각** 표시. 하루 1개만 선택 가능.

### 실시간 시계
- 프로토타입의 **데모 슬라이더는 제거**하고, 실제 현재 시각(KST)을 **1분마다 자동 갱신**하여 "지금 역할"이 흐르게 하라.

## 3. 데이터 모델 (명세서 8장 — PostgreSQL 테이블 그대로 구현)
명세서 8.1~8.7의 테이블을 마이그레이션 SQL로 작성하라:
`employees`, `daily_records`, `daily_number_assignments`, `attendance`, `substitute_plans`, `timetables`, `holidays`.
- 제약조건(PK, FK, UNIQUE) 명세대로. 출석 토글 = 행 삭제, 미출석 = 행 없음.
- RLS(Row Level Security)를 켜고, 익명 접근 허용하되 쓰기는 필요한 테이블에만 여는 정책을 작성하라.
- 6장 시간표를 `timetables`에, 프로토타입 공휴일 목록을 `holidays`에 시드하는 SQL 포함.

## 4. 화면 (명세서 10장 + 프로토타입 UI 이식)
1. **출석**(첫 화면): 7명 이름, 클릭 출석/토글, 미출석·대타 표시
2. **대시보드**: 지금 내 역할(대형), 내 번호·조·출석, 오늘 내 타임별 역할표, 전체 현황 그리드(미출석 표시), 날짜 자동판별 배지, (주말)바쁜 타임 선택+변경이력
3. **시간표**(조회용): 평일/주말 시간표 + 타임 분해 + 역할 흐름
4. **대타 캘린더**: 예약 등록/조회/삭제, 이력
프로토타입의 디자인 토큰(색 `C`, 레이아웃, 컴포넌트)을 유지하라. 깔끔한 업무용 대시보드 톤.

## 5. 작업 순서 (이 순서로 진행하고, 각 단계 끝에 간단히 보고)
1. Vite + React 프로젝트 초기화, 의존성 설치(@supabase/supabase-js 등), 폴더 구조 생성
2. 역할 계산·시간표·공휴일·날짜판별 로직을 프로토타입에서 `src/logic/`으로 이식 + 단위 테스트
3. Supabase 마이그레이션 SQL + 시드 + RLS 정책 작성 (`supabase/migrations/`)
4. supabase-js 클라이언트 설정, 환경변수(.env.example) 구성
5. 화면 4개 구현, 프로토타입 UI 이식, 데모 슬라이더→실시간 시계 교체
6. Realtime 구독 연결(출석·배정·바쁜 타임), 낙관적 업데이트
7. pg_cron 자정 배정 생성 함수 작성
8. README 작성: 로컬 실행법, Supabase 프로젝트 생성·키 발급·마이그레이션 적용법, Vercel 배포법, 환경변수 목록
9. 빌드 통과 확인(`npm run build`), 타입 에러 0 확인

## 6. 코딩 규칙
- 역할 계산 로직은 순수 함수로 분리하고 테스트로 정확성을 보장하라(프로토타입 검증 결과: 평일/주말 모두 항상 현장 3 고정, 1~N번 공정 휴식).
- 비밀키는 코드에 하드코딩 금지. `.env`/`.env.example`로 관리, `.gitignore` 포함.
- Supabase anon key는 프론트 노출이 정상이나, service_role 키는 절대 프론트에 두지 마라.
- 의미 있는 커밋 단위로 진행하라.
